# Alpha Internal Upgrade Audit V1

本文件是封閉測試前的內部升級審查。它延續 `docs/product-advisory-board.md` 的五位顧問框架，並加入「孫子 / 莊子 / 彼得杜拉克」三面鏡，檢查目前產品流程、頁面、元素、題庫與架構。

注意：本文件不是宣稱引用或複製任何真人顧問的私有方法，而是用公開可理解的產品、架構、UX、分析與成長框架來做內部決策。

## 參與角色

### 五位產品顧問

1. Marty Cagan 視角：產品價值與商業假設
2. Martin Fowler 視角：系統邊界、可演進架構與部署健康度
3. Don Norman 視角：使用者理解、操作提示與回饋
4. Avinash Kaushik 視角：漏斗、事件、意圖階段與可觀測性
5. Brian Balfour / Reforge 視角：分享、回訪與成長循環

### 付費報告三面鏡

1. 孫子兵法式策略視角：局勢、資源、勝算、避險與出手時機
2. 莊子式自我轉化視角：身份流動、自我鬆綁、順勢而為
3. 彼得杜拉克式管理視角：目標、成果、貢獻、時間與行動系統

### 六頂思考帽

- 白帽：現有事實與資料
- 紅帽：使用者感受與直覺

## 一、目前產品流程審查

### 現有流程

```text
入口頁
-> 測驗頁 6 題三選一
-> 結果揭曉
-> 完整分析
-> 淺度報告分享意圖 / 深度報告購買意圖
-> AI 好友圈 / 指南針好友圈
-> 未來導航會員頁
```

### 會議判斷

目前主流程已可支撐封閉測試，但仍偏「原型可跑」而非「轉換順暢」。最大的問題不是功能不足，而是每一步的產品意圖還沒有完全明確：

- 入口頁有神秘感，但需要更明確承諾「30 秒得到什麼」。
- 測驗頁已改善成三選一，但每題的選項圖還沒有真正到位。
- 結果頁身份揭曉感已建立，但「分享」「淺度報告」「深度報告」三個行動需要更清楚的層級。
- 好友圈已轉成指南針概念，但 mock data 與真實好友關係尚未分離乾淨。
- 未來導航頁目前仍像會員頁原型，還不是完整的付費前說服頁。

### 建議升級

1. 把整體流程改成三段意圖：
   - 免費身份揭曉：完成測驗、取得家徽、知道自己是哪一型。
   - 社交裂變：分享 2 位好友，解鎖淺度報告。
   - 付費深化：購買深度報告，進入指南針 / 每日陪伴。
2. 每段只保留一個主要 CTA，避免使用者在結果頁同時看到太多選擇。
3. 所有 CTA 都要寫入 `user_events`，Admin 要能看到每一步掉在哪裡。

## 二、頁面級審查

### 1. 入口頁

現況：

- 有 390px mobile app shell。
- 已加入九大家徽環繞與中央能量核心。
- CTA 是「開始探索」。

問題：

- 視覺記憶點強，但產品承諾略抽象。
- 桌面 / 平板目前仍以手機容器置中為主，正式版需要 stage layout，不要只像手機截圖。
- 中央動效要持續避免壓過 CTA，否則使用者會看動畫但不點。

建議：

- 補一行低字數承諾，例如「6 題看見你的 AI 原型與未來方向」。
- 入口頁事件應追蹤 `opened_app`、`started_quiz`。
- 未來 `apps/web` 正式化時，手機維持 app shell，桌面改成手機舞台加右側說明或測試 QR。

### 2. 測驗頁

現況：

- 6 題，每題 3 選 1。
- 選項已隨機排列。
- 前端 fallback 可用六維人格計算。
- API 可寫入 Supabase。

問題：

- 每個選項目前仍偏文字，缺少 18 張形象圖。
- 使用者理解「題目 vs 選項」已改善，但還需要視覺節奏：先看情境，再選反應。
- 選項被隨機後，A/B/C 是畫面位置代號，不是人格語意代號，資料記錄要持續保留原始 `optionId` 與 displayed position。

建議：

- Alpha-14 優先補 18 張選項圖 placeholder 結構，讓每個選項有圖像身份。
- Admin 題目圖片對照表下一步要能檢查每題 3 張圖是否缺失。
- 每題送出時記錄 `answered_question`，metadata 包含 `sceneId`、`optionId`、`displayIndex`、`dimensionEffect`。

### 3. 結果頁

現況：

- 已分 reveal / detail。
- 使用家徽作為身份揭曉。
- 有淺度報告分享意圖與深度報告購買意圖。

問題：

- 「預覽分享卡」「看完整分析」「分享 2 位好友」「我想買深度報告」之間層級仍偏多。
- 深度報告按鈕尚未說明會得到什麼，付費價值還不夠具體。
- 淺度報告目前只記錄 intent，未與好友邀請結果綁定。

建議：

- Reveal 第一屏只保留：
  - 主要：看完整分析
  - 次要：預覽分享卡
- Detail 區改成轉換區：
  - 免費：查看我的 AI 好友圈
  - 裂變：分享 2 位好友解鎖淺度報告
  - 付費：購買三面鏡深度報告
- 深度報告按鈕旁補短句：「策略、轉化、管理三個角度給你下一步」。

### 4. AI 好友圈 / 指南針好友圈

現況：

- 已有家徽牆 / 指南針社交概念。
- 已把付費或高意圖者推到指南針中心的產品方向寫入 Alpha-12。

問題：

- 現在仍大量 mock data。
- 好友焦慮的敘事很強，但要避免變成負面壓迫，需轉成「你可以追上」的行動感。
- `navigationStatus` 目前是 Alpha proxy，不是真付費狀態。

建議：

- 明確分四群：
  - 迷霧中：未測 / 未知
  - 已覺醒：完成測驗
  - 已校準：分享解鎖淺度報告
  - 指南針：購買或強烈付費意圖
- 朋友牆每個頭像要有可解釋狀態，而不是只有裝飾。
- 下一步要建立 `friend_links` 真實邀請鏈路與 `share_events`。

### 5. 未來導航會員頁

現況：

- 可從好友圈進入。
- 未接金流。

問題：

- 目前比較像「未來功能頁」，不是「購買前說服頁」。
- 尚未把深度報告和每日陪伴拆開。

建議：

- 把未來導航拆成兩層產品：
  - 一次性深度報告：三面鏡解讀 + 7/30/90 天行動。
  - 訂閱陪伴：每日指南針、任務、提醒、好友進度。
- Alpha 階段只記錄意圖，不收款。

### 6. Admin Dashboard

現況：

- 有登入。
- 有 Dashboard、用戶清單、測驗紀錄、題目圖片對照表、好友圈關係表。
- 已顯示六維分布、原型相似度、漏斗事件。

問題：

- `admin/admin123` 只能本機或封閉測試臨時使用，不可正式公開。
- Dashboard 仍偏資料列表，還不夠像「封測指揮中心」。
- 缺少健康紅燈：
  - API 是否可用
  - Supabase 寫入是否成功
  - 事件是否中斷
  - 哪一題掉最多
  - 哪個 CTA 沒人點

建議：

- Dashboard 第一屏改成：
  - 今日進站
  - 開始測驗
  - 完成測驗
  - 查看結果
  - 分享意圖
  - 深度報告意圖
  - API / DB 健康
- 題目圖片表增加 `missing asset` 警示。
- 用戶清單加入 `lastEvent` 與 `navigationStatus`。

## 三、元素級審查

### 家徽

保留。這是目前最強的品牌資產。線條樣式不可隨便改，只能改材質、使用情境、背景與動效。

升級方向：

- 家徽用途統一：
  - 入口：九大位置
  - 結果：我的身份
  - 好友圈：朋友分布
  - 分享卡：社交傳播
  - Admin：原型分布圖示

### 中央能量核心

保留，但只用在入口或重大揭曉，不要每頁都用，避免稀釋神秘感。

### CTA

目前主要 CTA 視覺足夠，但文案層級需要統一：

- 開始探索
- 看完整分析
- 分享 2 位好友解鎖淺度報告
- 我想買深度報告
- 查看我的 AI 好友圈
- 開啟未來導航

每頁只應有一個最亮 CTA。

### Toast / Modal

目前 toast 可用，但正式 LIFF 前要區分：

- 假功能提醒
- 意圖已記錄
- 分享成功 / 失敗
- 朋友解鎖進度

## 四、題庫審查

### 現況

- 6 題 × 3 選項。
- 18 個 option 已有 `dimensionEffect`。
- 分數不再直接對應原型，而是先累積六維，再與九大原型座標比對。

### 問題

- 6 題可以完成 Alpha 測試，但人格解析深度有限。
- 目前每個維度覆蓋次數不完全平均。
- 有些選項語意偏「理想自我」，使用者可能選想成為的人，不是當下真實反應。

### 建議

1. 封測先保留 6 題，避免流失。
2. 深度報告前可以加 2 題可選補充題：
   - 目前狀態：已決定 / 十字路口 / 停滯
   - 生日或星座：可選填，不要強迫
3. 正式版可做雙層題庫：
   - 免費：6 題，快、好分享
   - 深度：12-18 題，付費或分享解鎖後使用
4. 18 張選項圖必須從 `Option Bible` 推導，不要先畫圖再找人格解釋。

## 五、架構審查

### 現況

```text
.dc.html prototype
apps/web formal frontend shell
apps/api Railway API
packages/shared quiz engine and data
supabase migrations and schema
apps/admin operation console
Cloudflare Pages target frontend
Railway target API
Supabase target data layer
```

### 判斷

架構方向正確：前台、API、Admin、shared engine、資料層已分離。現在最大的技術債不是後端，而是前台仍以 `.dc.html` 原型為主要產品表面。

### 風險

- `.dc.html` 檔案巨大，長期維護成本高。
- 同一頁內 fallback、prototype runtime、正式流程並存，容易互相踩到。
- Admin 安全性不足，不可公開。
- LIFF 尚未接入，分享成長循環還未閉合。
- 事件追蹤若沒有 dashboard 紅燈，很難知道封測資料是否可信。

### 建議

1. 短期不重寫，先讓 `.dc.html` 撐過封測。
2. 封測後把最穩定的三頁逐步遷移到 `apps/web`：
   - 入口頁
   - 測驗頁
   - 結果頁
3. API 繼續作為唯一 privileged layer，不讓瀏覽器碰 Supabase secret。
4. Cloudflare 只放前端，Railway 只跑 API，Supabase 只存資料。

## 六、六頂思考帽結論

### 白帽：事實

- 本機前台流程已可跑。
- API 已能寫 Supabase。
- Admin 已能看測驗、用戶、事件、六維與好友圈資料。
- 付費與 LIFF 尚未真正接入。
- 題目圖片與分享卡仍未完全生產化。

### 紅帽：感受

- 家徽視覺有記憶點。
- 結果揭曉有身份感。
- 好友圈焦慮感有社交傳播潛力。
- 測驗如果沒有圖片，情緒沉浸仍不夠。

### 黑帽：風險

- 太早接付款會放大尚未驗證的體驗問題。
- `.dc.html` 長期不可維護。
- 管理後台公開前安全不足。
- 朋友關係如果是假資料，封測指標會誤導決策。

### 黃帽：機會

- 九大家徽可以成為品牌核心。
- 「分享 2 位好友解鎖淺度報告」有自然裂變機會。
- 深度報告三面鏡有清楚付費理由。
- 每日指南針可延伸訂閱陪伴。

### 綠帽：新方案

- 免費 6 題 + 可選生日 / 當下狀態，產生更豐富結果。
- 朋友牆改成「你的朋友有幾個已找到方向」。
- 深度報告先用 intent + 人工生成 MVP 驗證願付意願。
- Admin 變成封測作戰室。

### 藍帽：下一輪優先順序

1. Alpha-14：18 張選項圖接入與題目圖片健康檢查。
2. Alpha-15：封測漏斗作戰室，補齊事件紅燈。
3. Alpha-16：真實分享鏈路資料結構，先不接 LIFF 也能記錄 invite code。
4. Alpha-17：深度報告產品頁與 intent 後續流程。
5. Alpha-18：Cloudflare Pages 前端部署與 LIFF 準備。

## 七、立即值得優化的清單

### P0：封測前必做

- 確認入口、測驗、結果、好友圈、未來導航在手機 390px 無橫向捲動。
- 確認每個 CTA 都有事件。
- 確認 `/quiz/score` 寫入成功時 Admin 看得到新資料。
- 結果頁轉換區重新整理 CTA 層級。
- Admin 首頁加健康檢查與漏斗摘要。

### P1：封測中強烈建議

- 18 張選項圖接入。
- 分享卡視覺與文案整理。
- 好友圈 mock data 標記清楚，不與真資料混淆。
- 深度報告意圖後增加「留下通知方式」或「下一步說明」。

### P2：封測後

- 正式 LIFF login / shareTargetPicker。
- 真實 friend graph。
- 付款與會員權益。
- 每日指南針訂閱。
- `.dc.html` 逐步遷移到 `apps/web`。

## 八、內部升級建議結論

目前不是缺功能，而是需要把「視覺、資料、轉換、監控」接成閉環。

下一次升級不應該大改全部頁面，而應該先做：

```text
選項圖像化
-> 每題更有感
-> 完成率與答案品質提高
-> 結果更想分享
-> Admin 看見題目與轉換表現
```

建議下一輪正式命名為：

```text
Alpha-14 Option Image Integration And Funnel Health
```

目標：把 18 張選項圖放進測驗體驗，同時讓 Admin 能看出哪一題、哪一張圖、哪個 CTA 影響完成與轉換。
